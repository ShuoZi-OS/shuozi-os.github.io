# ShuoZi OS — 双盘架构

> 版本: v2.0 | 日期: 2026-05-20 | 取代三盘方案

---

## 设计

两个盘。消息互通。备份只存保留的。

```
┌─────────────────────────────┐    ┌─────────────────────────────┐
│          工作盘               │    │          备份盘               │
│       (1/2 总空间)           │    │       (1/2 总空间)           │
│                             │    │                             │
│  图形化 + AI 终端 共用        │    │  消息互通 ← 只接收保留事件     │
│                             │    │                             │
│  • 创建 / 编辑 / 删除        │    │  • 创建 ✓                   │
│  • 人和 AI 同盘操作          │    │  • 编辑 ✓                   │
│  • 可读写                   │    │  • 删除 ✗ (不传备份盘)        │
│                             │    │  • 全部压缩存储              │
│  不压缩（快速读写）           │    │  • 仅恢复时读取              │
└─────────────────────────────┘    └─────────────────────────────┘
```

---

## 消息流

```
事件: fs.write /a.txt "hello"     → 工作盘 ✓  → 备份盘 ✓
事件: fs.write /a.txt "world"     → 工作盘 ✓  → 备份盘 ✓
事件: fs.delete /a.txt            → 工作盘 ✓  → 备份盘 ✗ (不传)
```

---

## 容量分配

| 盘 | 占比 | 原因 |
|----|------|------|
| 工作盘 | 1/2 | 只存活跃文件，人和 AI 共用。消息互通，不需要两个独立工作盘 |
| 备份盘 | 1/2 | 存所有保留版本。压缩后一份备份盘能容纳几十倍工作盘的历史 |

备份盘一半空间看似多，实际：工作盘 500GB → 存 10 个活跃项目。备份盘 500GB × 压缩 ≈ 存几千次快照。够用。

---

## 误删恢复

```
AI 删了 /important/data.json
  → 工作盘：文件没了
  → 备份盘：data.json 还在（删除事件没传备份盘）

恢复：
  → 从备份盘读 data.json → 写回工作盘
  → 损失：零
```

---

## 为什么不是三个盘

三盘（图形化盘 + AI 终端盘 + 备份盘）的原因是想隔离人和 AI 的操作空间。

但实际上人和 AI 通过消息总线共享一切——AI 看得到人做了什么，人也看得到 AI 做了什么。分两个操作盘没有隔离效果，只多了一个无意义的边界。

两盘够用：干活的在一起，备份的自己一个。
